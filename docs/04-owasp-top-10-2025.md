# 04 - OWASP Top 10 2025 with Focus on APIs and Threat Modeling

This document presents the **OWASP Top 10 2025** with a practical perspective, focused on web applications and APIs, and how to use threat intelligence to prioritize and mitigate these risks more intelligently.

---

## Overview of the OWASP Top 10 2025

The OWASP Top 10 is the list of the most critical security risks for web applications and APIs, based on real data collected from thousands of applications.

### 2025 Ranking

| Rank | Risk | Category | Relevance for APIs | Typical Priority |
|------|------|----------|--------------------|------------------|
| **A01** | Broken Access Control | Access Control | Very High | ★★★★★ |
| **A02** | Cryptographic Failures | Cryptography | High | ★★★★☆ |
| **A03** | Injection | Injection | Very High | ★★★★★ |
| **A04** | Insecure Design | Insecure Design | High | ★★★★☆ |

## Detailed Analysis with Threat Intelligence Focus

### A01:2025 - Broken Access Control (Most Critical)

**Why is it the most dangerous in 2025?**
- It is the most exploited risk in multi-tenant APIs and modern applications.
- Many modern frameworks enable rapid development but leave authorization gaps if not used correctly.

**How Threat Intelligence helps:**
- **MITRE ATT&CK:** T1078 (Valid Accounts), T1134 (Access Token Manipulation), T1552 (Unsecured Credentials)
- During threat modeling, ask: "If an attacker obtains credentials from a low-privilege user, which resources can they access?"
- Monitor abnormal access patterns (e.g., a single token accessing thousands of resources in a short time)

**Priority Controls:**
- Server-side authorization on **all** endpoints
- Use of non-sequential UUIDs for resources
- Automated access control matrix testing

---

### A03:2025 - Injection

**Why it remains critical:**
- It is still one of the most direct ways to compromise the backend.
- GraphQL and NoSQL APIs have introduced new variations of injection attacks.

**How Threat Intelligence helps:**
- **MITRE ATT&CK:** T1190 (Exploit Public-Facing Application), T1059 (Command and Scripting Interpreter)
- Threat intelligence feeds can indicate which injection payloads are being used in active campaigns
- MISP can contain IOCs of automated tools exploiting injection flaws

**Priority Controls:**
- Parameterized queries / prepared statements
- Rigorous input validation
- WAF with updated rules (OWASP Core Rule Set)

---

### A05:2025 - Security Misconfiguration

**Why it is so common:**
- Default configurations of frameworks and cloud providers usually prioritize ease of use over security.
- Many teams lack a formal hardening process.

**How Threat Intelligence helps:**
- **SANS** and **CIS** publish alerts about configurations that are being actively exploited
- Talos and other feeds indicate when new misconfiguration exploitation techniques are observed

**Priority Controls:**
- Use **CIS Benchmarks** as the baseline for the entire stack
- Disable debug mode in production
- Remove headers that expose unnecessary information
- Automated hardening via Infrastructure as Code (IaC)

---

### A06:2025 - Vulnerable and Outdated Components

**Why it is critical in 2025:**
- Supply chain attacks are increasing (e.g., SolarWinds, XZ Utils, Codecov)
- The volume of published CVEs continues to grow

**How Threat Intelligence helps:**
- **NVD + EPSS** helps prioritize CVEs that are actually being exploited
- MISP and Talos can indicate when a specific CVE is being used in active campaigns
- Supply chain attack intelligence helps understand broader risks

**Priority Controls:**
- Automated SCA in the CI/CD pipeline
- SBOM generation on every release
- Prioritization based on CVSS + EPSS + active exploitation context

---

### A08:2025 - Software and Data Integrity Failures

**Why it gained importance:**
- Attacks on supply chains and CI/CD pipelines are becoming more common and impactful.

**How Threat Intelligence helps:**
- **MITRE ATT&CK:** T1195 (Supply Chain Compromise), T1608 (Stage Capabilities)
- Threat intelligence on supply chain attacks helps prioritize integrity controls
- Monitor repositories and packages for suspicious behavior

**Priority Controls:**
- Code signing for important artifacts
- SBOM + provenance attestation
- Strong hardening of the CI/CD pipeline

## How to Prioritize the OWASP Top 10 in Your Context

Not all risks are equally important for every application.

### Prioritization Recommendation

| Application Type                      | Highest Priority Risks      | Justification |
|---------------------------------------|-----------------------------|-------------|
| **Public / Multi-tenant API**         | A01, A03, A07, A05          | Larger attack surface + high impact of unauthorized access |
| **Internal Corporate Web App**        | A05, A06, A02               | Lower external exposure, but high risk of misconfiguration and vulnerable components |
| **Microservices / Cloud Native**      | A01, A06, A08               | Complexity of inter-service communication + supply chain risks |
| **Highly Sensitive Data Application** | A02, A01, A07, A09          | Data leakage impact is critical |

## Integrating OWASP Top 10 with MITRE ATT&CK

This is one of the most powerful parts of the framework.

| OWASP Risk                        | Most Relevant ATT&CK Techniques | Threat Modeling Question |
|-----------------------------------|---------------------------------|--------------------------|
| A01 - Broken Access Control       | T1078, T1134, T1552             | "How could an attacker with valid low-privilege credentials escalate access?" |
| A03 - Injection                   | T1190, T1059                    | "How could an attacker inject code through the inputs of this API?" |
| A05 - Security Misconfiguration   | T1059, T1608                    | "Which default configurations of this framework/cloud are exposed and how would an attacker exploit them?" |
| A06 - Vulnerable Components       | T1195                           | "If this dependency is compromised, what would be the impact?" |
| A07 - Authentication Failures     | T1078, T1110, T1552             | "How could an attacker obtain valid credentials or bypass authentication?" |
| A08 - Software Integrity          | T1195, T1608                    | "How could an attacker compromise our pipeline or release artifacts?" |

## Practical Recommendation

1. **Do not treat the OWASP Top 10 as a generic checklist.** Use it as a starting point for risk discussions in your specific context.
2. **Always combine it with MITRE ATT&CK** during threat modeling. This makes the exercise much more concrete.
3. **Use threat intelligence** to prioritize which risks deserve more attention right now (which ones are being actively exploited).
4. **Document** your prioritization decisions and implemented controls.

## Connection with the Integrated Secure Development Model

The OWASP Top 10 should be used as **input** in the **Prepare** and **Design** phases of NIST SSDF, and as a **verification checklist** in the **Produce** and **Deploy** phases.

| SSDF Phase     | How to Use OWASP Top 10 + Threat Intel                                                              | Practical Example |
|----------------|-----------------------------------------------------------------------------------------------------|-------------------|
| **Prepare**    | Define Risk Appetite + prioritize risks based on active threat intelligence                         | "In public APIs, Broken Access Control (A01) has maximum priority due to active BOLA campaigns" |
| **Design**     | Threat modeling using OWASP risks + relevant ATT&CK techniques                                      | Map A01 → T1078 + T1134 during endpoint design |
| **Produce**    | Code review + automated tests focused on priority risks                                             | Authorization tests + SCA for vulnerable components |
| **Deploy**     | Security gates in the pipeline + runtime protection (e.g., FastAPI Threat Shield)                   | Block builds with critical CVEs + threat intel middleware |
| **Respond**    | Structured logging + correlation with threat intel IOCs                                             | Detect exploitation patterns of A01/A03 in production |

## Lessons from Daniel Lemeszenski's Course (Impacta Digital)

From the materials on **API Security** and **Governance**:

- **"Less is More"**: Limit the information returned and permissions granted. This directly mitigates A01 (Broken Access Control) and A05 (Security Misconfiguration).
- **Double Custody**: Both backend and frontend (mobile) must protect the application. Even if one layer fails, the other must compensate.
- **BOLA Prevention**: Always validate whether the authenticated user has permission to access the requested object (compare the `user_id` from the token with the one in the URL/parameter).
- **Improper Inventory Management (API Risk #9)**: Control environments (dev/QA/prod), segregate sensitive data, and keep documentation updated.

These practical principles perfectly complement the OWASP Top 10 and should be applied throughout the entire development lifecycle.
